# Mizuta MAP 仕様書

## 1. 概要

Mizuta MAP は、「水たまり」をテーマにしたスマートフォン向けマップアプリである。
地図上に水たまりの投稿をピンとして表示し、ユーザーは水たまりの大きさ、透明度、観測日時などを確認できる。
投稿機能ではカメラで水たまりを撮影し、必要項目を入力してマップ上のピンに変換する。
AR 機能では、スマホカメラ越しに水たまりの上へ魚を表示する。

## 2. 目的

- 水たまりを地図上の POI として可視化する
- 投稿された水たまりを 1 週間だけ地図上に表示する
- 水たまりの状態を簡単に記録、共有できるようにする
- AR 表現により、水たまりを遊びや発見につなげる

## 3. 想定ユーザー

- 身近な水たまりを見つけて記録したい人
- 雨の日の街歩きや観察を楽しみたい人
- 地図上で水たまりの場所や状態を確認したい人

## 4. 画面構成

アプリは下部フッターで 3 画面を切り替える。

- HOME
- 投稿
- AR

## 5. HOME 仕様

### 5.1 役割

水たまり投稿を地図上で確認するメイン画面。
OpenStreetMap を使用し、水たまりの位置をピンで表示する。

### 5.2 表示内容

地図上のピンには、水たまり投稿を表示する。
ピンを押すと、詳細パネルを表示する。

ピン詳細に表示する項目:

- 大きさ
- 透明度
- 観測日時

表示しない項目:

- 座標、緯度経度

### 5.3 表示期間

投稿された水たまりは、観測日時から 1 週間だけ HOME の地図上に表示する。
1 週間を過ぎた投稿は、地図上のピンとして表示しない。

### 5.4 検索機能

HOME には絞り検索を設置する。
スマホ画面を圧迫しないよう、検索 UI はアイコンボタンから開く形式にする。

絞り込み候補:

- 大きさ
- 透明度
- 観測日時
- 現在地からの距離

初期版では、大きさ、透明度、観測日時の 3 条件を優先する。

### 5.5 Google Map 経路連携

ピン詳細内に経路ボタンを置く。
経路ボタンを押すと、該当ピンの座標を目的地として Google Map を開く。

ユーザーには緯度経度の文字列は表示しない。
内部処理では投稿データの座標を利用する。

想定URL:

```text
https://www.google.com/maps/dir/?api=1&destination=緯度,経度
```

## 6. 投稿 仕様

### 6.1 役割

ユーザーが水たまりを撮影し、ピンに表示する項目を入力して投稿する画面。

### 6.2 入力フロー

1. カメラを起動する
2. 水たまりを撮影する
3. ピンに表示する項目を入力する
4. 投稿ボタンを押す
5. 投稿内容を地図上のピンに変換する
6. HOME に戻り、投稿したピンを表示する

### 6.3 入力項目

必須項目:

- 写真
- 大きさ
- 透明度
- 観測日時
- 座標

任意項目:

- コメント
- 水深
- 天気

座標はユーザーに直接入力させず、スマホの位置情報から取得する。

### 6.4 投稿ボタン

投稿ボタンを押すと、入力内容を POI データとして保存する。
保存されたデータは HOME の地図上でピンとして表示する。

初期版では、端末内保存またはローカルデータとして扱う。
複数ユーザーで共有する正式版では、サーバーまたはデータベース保存が必要になる。

## 7. AR 仕様

### 7.1 役割

スマホカメラを起動し、水たまりの上に魚を表示する画面。

### 7.2 表示内容

- カメラ映像を全画面表示する
- 画面上に魚の 3D 表現または 3D 風表現を重ねる
- フッターから HOME / 投稿 / AR に戻れるようにする

### 7.3 初期版の実装方針

初期版では、水たまり自動認識までは行わず、画面中央またはタップ位置に魚を表示する。
正式版では、画像認識や平面検出を利用して、水たまり位置に合わせた表示を検討する。

## 8. フッター仕様

画面下部に固定フッターを配置する。

項目:

- HOME
- 投稿
- AR

HOME は地図アイコン、投稿はカメラまたは追加アイコン、AR は魚またはカメラアイコンで表現する。
文字は最小限にし、スマホで押しやすいサイズにする。

## 9. データ仕様

### 9.1 投稿データ

1 件の投稿は、以下の情報を持つ。

```json
{
  "id": "post-001",
  "lat": 33.9743512,
  "lng": 134.3612004,
  "size": 242,
  "transparency": 5,
  "observedAt": "2026-07-15T10:00:00+09:00",
  "image": "path-or-url",
  "comment": "",
  "createdAt": "2026-07-15T10:05:00+09:00"
}
```

### 9.2 GeoJSON 変換

HOME の地図表示では、投稿データを GeoJSON の Point として扱う。

```json
{
  "type": "Feature",
  "geometry": {
    "type": "Point",
    "coordinates": [134.3612004, 33.9743512]
  },
  "properties": {
    "大きさ": 242,
    "透明度": 5,
    "観測日時": "2026-07-15T10:00:00+09:00",
    "画像": "path-or-url"
  }
}
```

## 10. 3人分業の進め方

3人で分担する場合は、各担当が自由に画面を作るのではなく、先に「接続要素」を共通化する。
接続要素とは、画面名、データ形式、保存場所、共通関数、画面切り替えルールのことである。
この部分が揃っていれば、HOME、投稿、AR を別々に作っても最後に統合しやすい。

### 担当A: HOME / 地図担当

担当範囲:

- OpenStreetMap 表示
- 水たまりピン表示
- ピン詳細パネル
- 1週間だけ表示する処理
- 検索、絞り込み
- Google Map 経路連携

成果物:

- `index.html` の HOME 画面
- `app.js` の地図、ピン、検索処理
- ピン詳細 UI

### 担当B: 投稿 / データ担当

担当範囲:

- カメラ起動
- 投稿フォーム
- 位置情報取得
- 投稿ボタン
- 投稿データ保存
- 投稿データから GeoJSON への変換

成果物:

- 投稿画面
- 入力フォーム
- 投稿保存処理
- HOME に渡す投稿データ

### 担当C: AR / UI統合担当

担当範囲:

- AR 画面
- 魚の表示
- フッター UI
- 画面切り替え
- 全体デザイン調整
- スマホ表示確認

成果物:

- AR 画面
- フッター
- 画面遷移
- レスポンシブ調整

## 11. 3人の接続要素

### 11.1 共通画面名

画面切り替えに使う名前は、以下で統一する。

```text
home
post
ar
```

各担当は、自分の画面だけを作る場合でも、この画面名で呼び出せるようにする。

例:

```js
showScreen("home");
showScreen("post");
showScreen("ar");
```

### 11.2 共通HTML構造

3画面は、同じ `index.html` 内に画面セクションとして置く。
画面の表示、非表示は JavaScript で切り替える。

```html
<main id="app">
  <section id="screen-home" data-screen="home"></section>
  <section id="screen-post" data-screen="post" hidden></section>
  <section id="screen-ar" data-screen="ar" hidden></section>

  <footer id="app-footer">
    <button data-nav="home">HOME</button>
    <button data-nav="post">投稿</button>
    <button data-nav="ar">AR</button>
  </footer>
</main>
```

担当Aは `screen-home`、担当Bは `screen-post`、担当Cは `screen-ar` と `app-footer` を中心に作業する。

### 11.3 共通データ形式

投稿画面から HOME 画面へ渡すデータは、以下の形に統一する。

```js
{
  id: "post-001",
  lat: 33.9743512,
  lng: 134.3612004,
  size: 242,
  transparency: 5,
  observedAt: "2026-07-15T10:00:00+09:00",
  image: "",
  comment: "",
  createdAt: "2026-07-15T10:05:00+09:00"
}
```

各項目の意味:

- `id`: 投稿ごとの一意なID
- `lat`: 緯度。ユーザーには表示しない
- `lng`: 経度。ユーザーには表示しない
- `size`: 水たまりの大きさ
- `transparency`: 透明度
- `observedAt`: 観測日時。1週間表示の判定に使う
- `image`: 撮影画像
- `comment`: 任意コメント
- `createdAt`: 投稿日時

### 11.4 共通保存場所

初期版では、投稿データを `localStorage` に保存する。
保存キーは以下で統一する。

```text
mizutaPosts
```

投稿担当は `mizutaPosts` に保存し、HOME 担当は `mizutaPosts` から読み込む。
別々に作業する場合でも、保存キーを変えない。

### 11.5 共通関数名

画面間の接続で使う関数名は、以下で統一する。

```js
showScreen(screenName);
loadPosts();
savePost(post);
renderPins(posts);
refreshHome();
openGoogleMaps(lat, lng);
startCamera(mode);
stopCamera();
```

関数の役割:

- `showScreen(screenName)`: HOME、投稿、AR を切り替える
- `loadPosts()`: 保存済み投稿データを読み込む
- `savePost(post)`: 投稿データを保存する
- `renderPins(posts)`: 投稿データを地図ピンとして表示する
- `refreshHome()`: HOME のピン表示を更新する
- `openGoogleMaps(lat, lng)`: Google Map の経路を開く
- `startCamera(mode)`: 投稿または AR でカメラを起動する
- `stopCamera()`: 画面移動時にカメラを停止する

### 11.6 画面遷移ルール

フッターを押したときの動きは以下に統一する。

```text
HOME を押す
  -> カメラを停止する
  -> HOME を表示する
  -> ピンを再描画する

投稿 を押す
  -> AR のカメラを停止する
  -> 投稿画面を表示する
  -> 投稿用カメラを起動する

AR を押す
  -> 投稿用カメラを停止する
  -> AR 画面を表示する
  -> AR 用カメラを起動する
```

### 11.7 投稿完了時の接続ルール

投稿ボタンを押した後は、以下の順番で処理する。

```text
入力値を取得
  -> 現在地を取得
  -> 投稿データを作成
  -> savePost(post) で保存
  -> showScreen("home") で HOME に戻る
  -> refreshHome() でピンを再表示
```

この流れを守ることで、投稿担当が作った内容を HOME 担当の地図に接続できる。

### 11.8 ピン押下時の接続ルール

HOME のピンを押すと、詳細パネルを表示する。
詳細パネルでは、座標は表示しない。
ただし、経路ボタンを押したときだけ内部的に座標を使う。

```text
ピンを押す
  -> 大きさ、透明度、観測日時を表示
  -> 経路ボタンを表示

経路ボタンを押す
  -> openGoogleMaps(lat, lng)
  -> Google Map を開く
```

### 11.9 AR 画面の接続ルール

AR 画面はカメラを使うため、他画面と接続する際にカメラ停止処理が必要になる。

```text
AR に入る
  -> startCamera("ar")
  -> 魚を表示

AR から出る
  -> stopCamera()
  -> 選択された画面へ移動
```

AR 担当は、フッター移動時にカメラが止まることを必ず確認する。

### 11.10 結合時の確認項目

3人の作業を結合するときは、以下を確認する。

- フッターから HOME / 投稿 / AR に移動できる
- 投稿後に HOME へ戻る
- 投稿した内容が HOME のピンとして表示される
- ピン詳細に大きさ、透明度、観測日時だけが表示される
- 緯度経度は画面に表示されない
- 経路ボタンから Google Map が開く
- AR から別画面へ移動したときにカメラが止まる
- 1週間を過ぎた投稿が HOME に表示されない
- 検索、絞り込み後も投稿ピンが正しく表示される

## 12. 共通ルール

- まずスマホ画面を優先して作る
- 画面上の文字は最小限にする
- 座標は内部的に使うが、ユーザーには表示しない
- データ項目名はチーム内で統一する
- 1つの機能を作ったら、必ず HOME / 投稿 / AR のどこに影響するか確認する
- 大きな変更をする前に、他の2人へ共有する
- `data-screen`、`data-nav`、`mizutaPosts`、共通関数名は勝手に変更しない
- 各担当は、自分の画面だけで完結させず、接続先の関数を呼ぶところまで作る

## 13. 開発順序

1. 共通HTML構造とフッターを作る
2. `showScreen()` で HOME / 投稿 / AR を切り替えられるようにする
3. `loadPosts()`、`savePost()` の保存処理を作る
4. HOME の地図と既存ピン表示を完成させる
5. 投稿画面で入力項目を作る
6. 投稿内容を HOME のピンに変換する
7. 1週間だけ表示する処理を入れる
8. 検索、絞り込みを追加する
9. Google Map 経路連携を追加する
10. AR 画面で魚を表示する
11. スマホで全体確認する

## 14. 初期版で作る範囲

初期版では、以下を完成目標とする。

- HOME で水たまりピンを表示
- ピン詳細に大きさ、透明度、観測日時を表示
- 座標は表示しない
- 観測日時から 1 週間以内のピンだけ表示
- 検索、絞り込み
- 投稿フォーム
- 投稿ボタンでピン追加
- AR 画面で魚を表示
- フッターで3画面を切り替え
- Google Map 経路ボタン
- 投稿、HOME、AR が共通関数で接続されている

## 15. 判断が必要な点

1. 「透明度」は 1〜5、1〜10、または「透明 / 少し濁り / 濁り」のどれで入力するか
2. 「大きさ」は直径 cm で統一するか、縦横 cm で入力するか
3. 投稿データは初期版では端末内保存でよいか、それとも複数人で共有できる保存方法が必要か
4. AR の魚は、初期版では画面中央固定でよいか、タップ位置に出すか
5. 1週間表示の基準は「観測日時」か「投稿日時」か
